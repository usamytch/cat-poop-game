# Cat Poop Game — browser performance baseline

Дата замера: **12 июля 2026**

Версия: два среза — **до и после внедрения fixed timestep**, до новых игровых систем

## Зачем этот документ

Это локальная точка отсчёта для последующих before/after-замеров. Она отвечает на два разных вопроса:

1. Укладываются ли текущие логика и Canvas-рендер в кадровый бюджет?
2. Одинаково ли быстро идёт игровое время на экранах с разной частотой обновления?

Ответы оказались разными: запас CPU очень большой, но игровое время сейчас зависит от частоты `requestAnimationFrame`.

## Методика

- Игра запущена через `python3 -m http.server 8080`.
- Замер выполнен в отдельной чистой вкладке встроенного Chromium-браузера Codex после закрытия мешавшей браузерной сессии.
- Логический Canvas: `1200×700`; видимая область браузера: `1280×720`.
- Частота браузера во всех сценах: около **120 Гц**.
- Каждая сцена измерялась непрерывно **5 секунд**, примерно по 600 кадров.
- `frameMs` — интервал между соседними callback `requestAnimationFrame`.
- `cpuMs` — синхронное время полного `update() + draw()` по `performance.now()`.
- `frames > 20 ms` и `frames > 33.4 ms` — счётчики заметных длинных кадров.
- Сборщик выключен по умолчанию и включается клавишей `F`; его небольшой собственный overhead входит во все сравниваемые результаты одинаково.

Это честный локальный browser baseline, но не замена будущим замерам на массовом ноутбуке и реальном Android-устройстве.

### Что видно по цифрам

- Даже самая тяжёлая из снятых сцен использует на P95 лишь **0.5 ms CPU** на полный `update() + draw()`.
- Panic примерно удваивает CPU-стоимость относительно стартового экрана, но абсолютная цена остаётся очень низкой.
- Погоня с A* и обе разновидности Подвала не создают устойчивых длинных кадров.
- Единственный заметный CPU-пик — **4.1 ms** на DFS-карте; он не сорвал выдачу кадра и требует наблюдения, а не преждевременной оптимизации.
- Ни в одной из шести сцен не было кадра длиннее 20 ms.

## Главный найденный дефект: 120 FPS означает 120 simulation steps

Текущий цикл вызывает один `update()` на каждый `requestAnimationFrame`, а механика выражена в величинах «за кадр при 60 FPS». Поэтому на 120-герцовом экране:

- срочность растёт примерно вдвое быстрее реального замысла;
- кот, хозяин, снаряды и движущиеся препятствия обновляются примерно вдвое чаще;
- кадровые таймеры, cooldown, паника и длительности части эффектов идут не в едином реальном времени;
- одинаковый забег ощущается и балансируется по-разному на 60, 120 и 144 Гц.

Это не нехватка производительности — наоборот, браузер легко выдаёт 120 FPS. Это дефект игрового времени, который становится заметен именно благодаря хорошей производительности.

## Реализованное решение

Внедрён fixed simulation timestep **60 Гц** с независимой отрисовкой:

- `requestAnimationFrame` остаётся единственным драйвером рендера;
- накопитель реального времени запускает `update()` шагами по `1000/60 ms`;
- после зависания delta ограничивается;
- за один render frame выполняется не более 5 simulation steps;
- лишний backlog отбрасывается, чтобы не получить spiral of death;
- таймеры и игровая скорость сохраняют нынешнее 60 FPS-поведение;
- 120-герцовый экран получает 120 отрисовок, но ровно 60 шагов механики в секунду;
- таймер `levelMessageTimer`, ранее ошибочно изменявшийся в `drawUI()`, перенесён в `update()`.

## Повторный замер после fixed timestep

Условия и длительность те же. Широкий проход выполнен по всем шести сценам; затем финальная версия benchmark со встроенным счётчиком simulation steps повторно проверена на обычном уровне и DFS.

| Сцена | FPS | Simulation Hz | Frame P95 | CPU P95 | CPU max | >20 / >33.4 ms |
|---|---:|---:|---:|---:|---:|---:|
| Стартовый экран, Normal | 119.98 | 60* | 9.3 ms | 0.3 ms | 0.3 ms | 0 / 0 |
| Normal L1, спокойная сцена без хозяина | 120.00 | **60.10** | 9.3 ms | 0.3 ms | 0.5 ms | 0 / 0 |
| Normal L1, panic >75% | 119.98 | 60* | 9.3 ms | 0.4 ms | 0.7 ms | 0 / 0 |
| Chaos L1, погоня с хозяином | 120.00 | 60* | 9.3 ms | 0.4 ms | 0.9 ms | 0 / 0 |
| Normal L9, Подвал corridor | 119.90 | 60* | 8.8 ms | 0.4 ms | 0.6 ms | 0 / 0 |
| Normal L20, Подвал DFS | 120.09 | **60.15** | 9.2 ms | 0.5 ms | 0.7 ms | 0 / 0 |

`60*` — частота гарантируется тем же единым игровым циклом и тестами 60/120/144 Гц; прямой браузерный счётчик был добавлен после широкого прохода и отдельно подтверждён на обычной и самой тяжёлой DFS-сцене.

### Before/after вывод

- Частота рендера сохранилась на уровне 120 FPS.
- Симуляция теперь выполняет около 60,1 шага/с вместо одного шага на каждый из примерно 120 rAF.
- Normal L1 достигает порога паники примерно за 39,5–40 реальных секунд вместо примерно 20 секунд на 120 Гц.
- CPU P95 не ухудшился: старт и обычная сцена сохранили 0.3 ms; panic улучшился с 0.5 до 0.4 ms; corridor — с 0.5 до 0.4 ms; DFS остался на 0.5 ms.
- Единичный DFS-пик снизился с 4.1 до 0.7 ms в финальном повторе.
- До и после изменения не было ни одного кадра длиннее 20 ms.
- Unit/integration suite после изменения: **570/570 тестов**.

Проходной gate выполнен:

- игровая скорость на 60/120/144 Гц совпадает;
- FPS и frame pacing не хуже исходного baseline;
- CPU P95 не регрессировал;
- нет кадров >20 ms в локальных коротких сценах;
- unit/integration suite полностью зелёный.

## Проверка паузы после fixed timestep

Browser smoke-test выполнен в том же Chromium-профиле на 120 Гц:

- ручная пауза по `P` показала отдельный overlay поверх текущего мира;
- срочность оставалась ровно **37%** в начале и после **4.2 секунды** паузы;
- позиции кота, бонуса и геометрии на двух снимках совпали;
- после явного продолжения по `Enter` срочность стала **39%** через 1.1 секунды — скрытые 4.2 секунды не были доначислены;
- браузерная консоль не зарегистрировала ошибок;
- unit-тесты отдельно подтверждают заморозку `lifeLost`, бонусных таймеров, `simulationTimeMs` и движущихся препятствий, сброс accumulator, auto-pause по `blur`/`visibilitychange` и сохранение Web Audio clock.

После добавления паузы полный suite: **578/578 тестов**.

## Что ещё не измерено

Для полного Performance Contract отдельно нужны:

- массовый ноутбук с интегрированной графикой в актуальных Chrome и Firefox;
- реальный Android среднего класса в landscape;
- input latency «событие → simulation tick → видимый кадр»;
- отдельные P95 для A* repath и генерации уровня;
- stress-сцена с массовыми частицами, снарядами, бонусами и движущимися препятствиями одновременно;
- 10- и 30-минутный memory/audio soak-test;
- влияние DPR 2×/3×/4× после внедрения корректного high-DPI backing store.

Локальный baseline нельзя использовать как доказательство скорости на всех устройствах. Его назначение — честно поймать регрессию на той же машине и зафиксировать найденную зависимость симуляции от refresh rate.

### Шаг 4 — Проверка

- Убедиться, что изменения не ломают другие модули
- Проверить совместимость с существующими механиками (комбо, паника, сложности)
- Если есть тесты — сверить с [`plans/test-plan.md`](plans/test-plan.md)
- Если изменения затрагивают геймплей, управление, HUD, игровой цикл или визуальную
  обратную связь, обязательно выполнить smoke-test во встроенном браузере Codex:
  пройти изменённый пользовательский сценарий, проверить управление и ключевые
  переходы состояний. Одной успешной загрузки страницы недостаточно.
- Для каждого изменённого HUD/overlay-состояния сделать и визуально проверить
  screenshot в реальном размере Canvas. Текст кнопок, карточек и панелей должен
  полностью помещаться внутри границ, иметь заметные внутренние поля и не
  пересекаться с соседними элементами. Особенно проверять длинные русские строки,
  финальные CTA и mobile/touch-вариант; отсутствие console errors не заменяет эту проверку.
- Между отдельными элементами интерфейса держать минимум `16px` визуального
  воздуха; внутри одной компактной HUD-группы — минимум `12px`. Считать зазор
  между реальными границами текста/контрола, а не между baseline/центрами, и
  проверять его при максимальном pulse/scale анимации. Если вторичная подсказка
  дублирует очевидное действие и не помещается с этим зазором — убирать её.

## 🔑 Ключевые правила из README.md, которые агент должен помнить

### Игровые механики
- Шкала срочности растёт каждый кадр: `urgeRate/60 * (1 + (level-1)*0.08)`
- Паника при `urge/maxUrge > 0.75`
- Авария при `urge >= maxUrge`
- Комбо при 3 попаданиях подряд → `owner.flee()`
- `hitUrgeReduce` всегда меньше прироста за кулдаун (нельзя читерить стрельбой)

### Режимы
| Режим | Ключ | Правило |
|---|---|---|
| 🎓 Обучение | `tutorial` (`gameMode`) | 3 фиксированных экрана, Normal-физика, без статистики |
| 😼 Нормал | `normal` | Хозяин с 2 уровня |
| 😈 Хаос | `chaos` | Хозяин с 1 уровня |

`difficulty` содержит только `normal`/`chaos`; обучение выбирается через `gameMode` и не является скрытым балансным профилем.

### Бонусы
| Бонус | Ключ | Эффект |
|---|---|---|
| 🐟 Рыбка | `fish` | Ускорение x1.7, 5 сек |
| 🧶 Клубок | `yarn` | Хозяин замирает, 5 сек |
| 💊 Таблетка | `pill` | -30%/-40%/-50% срочности по фазам игры |
| ❤️ Жизнь | `life` | +1 жизнь, максимум 9 |
| 🌿 Котовник | `catnip` | Хозяин уходит в угол и игнорирует кота, 10 сек |

### Технические ограничения
- Только Vanilla JS — никаких фреймворков и npm-пакетов
- Весь звук — процедурный через Web Audio API
- Вся графика — Canvas 2D
- Прогресс — только через `localStorage`
- Запуск: открыть [`index.html`](index.html) в браузере (или `python3 -m http.server 8080`)

## 🗂️ Работа с git при манипуляциях с файлами

При добавлении, удалении или переименовании файлов агент ОБЯЗАН использовать git-команды, чтобы система контроля версий корректно отслеживала изменения.

### Переименование / перемещение файла

```bash
git mv старый/путь/файл.md новый/путь/файл.md
```

Пример: переименовать `plans/agent-plan.md` в `AGENTS.md`:
```bash
git mv plans/agent-plan.md AGENTS.md
```

> ⚠️ Не создавай новый файл + удаляй старый вручную — git потеряет историю. Всегда используй `git mv`.

### Добавление нового файла

```bash
git add путь/к/новому/файлу.md
```

Пример: добавить новый план:
```bash
git add plans/new-feature-plan.md
```

> После создания файла через `write_to_file` он появляется как `??` (untracked). Нужно явно добавить через `git add`.

### Удаление файла

```bash
git rm путь/к/файлу.md
```

Пример: удалить устаревший план:
```bash
git rm plans/old-plan.md
```

> Не удаляй файлы через `rm` или файловый менеджер — используй `git rm`, чтобы удаление попало в индекс.

### Проверка статуса

```bash
git status --short
```

Расшифровка статусов:
- `M` — изменён (modified)
- `A` — добавлен (added/staged)
- `R` — переименован (renamed)
- `D` — удалён (deleted)
- `??` — не отслеживается (untracked) — нужен `git add`
